# 配置历史与回滚实施计划

## 背景

当前管理端已经具备运营配置编辑能力，并在上一阶段补齐了操作日志和批量审核预览。下一阶段应继续完善“管理端协作与审计增强”的闭环，把配置修改从“可记录”推进到“可回看、可对比、可恢复”。

本计划先聚焦 `app_configs`，覆盖 `miniapp.home`、`quality.rules`、`data_source.defaults` 等现有运营配置。指标配置、首页推荐配置和质量规则表单化可以在该机制稳定后复用同一套版本能力。

## 阶段目标

- 配置每次保存都有历史版本。
- 配置提交前可以看到 diff 和影响摘要。
- 配置误改后可以从历史版本恢复。
- 配置保存、预览和回滚都能写入操作日志。
- 管理端运营人员可以在页面上完成历史查看和回滚确认。

## 工作项一：配置历史模型

### 目标

为运营配置建立独立版本记录，避免只依赖当前值和通用操作日志，保证配置变更可以长期追踪和恢复。

### 实施内容

- 新增 `config_versions` 或等价模型。
- 记录配置 key。
- 记录版本号。
- 记录变更前配置值。
- 记录变更后配置值。
- 记录操作人。
- 记录变更原因。
- 记录创建时间。
- 保存配置时自动生成新版本。

### 建议交付物

- 配置版本 SQLAlchemy model。
- Alembic migration。
- 配置版本 repository/service。
- 配置保存链路接入版本写入。
- 配置版本写入测试。

### 验收标准

- 修改 `miniapp.home` 会生成一条配置版本。
- 修改 `quality.rules` 会生成一条配置版本。
- 修改 `data_source.defaults` 会生成一条配置版本。
- 配置版本包含操作人、版本号、前后值和时间。
- 配置版本写入失败时不应静默吞掉错误。

## 工作项二：配置 History / Diff API

### 目标

让管理端能够查看某个配置项的历史版本，并在提交或回滚前展示清晰的差异。

### 实施内容

- 新增配置历史列表 API：
  - `GET /admin/configs/{key}/versions`
- 新增配置历史详情 API：
  - `GET /admin/configs/{key}/versions/{version_id}`
- 新增配置 diff 预览 API：
  - `POST /admin/configs/{key}/preview`
- diff 返回新增字段、删除字段、修改字段。
- diff 返回旧值、新值和字段路径。
- 对复杂 JSON 配置提供摘要，例如推荐指标数量、推荐城市数量、质量规则数量。

### 建议交付物

- 配置版本响应 schema。
- 配置 diff 响应 schema。
- JSON diff helper。
- 配置历史和 diff API。
- API 测试。

### 验收标准

- 可以查询指定配置 key 的历史版本列表。
- 可以查询指定历史版本详情。
- 提交新配置前可以得到 diff 预览。
- diff 能表达新增、删除和修改字段。
- 非法配置 key 返回明确错误。

## 工作项三：配置回滚 API

### 目标

支持将当前配置恢复到历史版本，同时保留回滚动作本身的版本记录和操作日志。

### 实施内容

- 新增配置回滚 API：
  - `POST /admin/configs/{key}/rollback`
- 回滚参数包含目标版本 ID。
- 回滚前复用 diff 逻辑生成当前值与目标版本的差异。
- 回滚成功后更新 `app_configs.value`。
- 回滚成功后生成新的 `config_versions` 记录。
- 回滚成功后写入 `operation_logs`。
- 回滚失败时返回明确原因。

### 建议交付物

- 配置回滚 service。
- 配置回滚 API。
- 回滚操作日志接入。
- 回滚测试。

### 验收标准

- 可以把 `miniapp.home` 回滚到历史版本。
- 回滚后当前配置值与目标历史版本一致。
- 回滚动作本身会生成新的配置版本。
- 回滚动作会写入操作日志。
- 不存在的版本 ID 不能回滚。

## 工作项四：管理端配置中心增强

### 目标

让运营人员不需要直接查 API，就能在管理端查看配置历史、预览差异并执行回滚。

### 实施内容

- 在“运营配置”卡片增加“历史”入口。
- 展示最近配置版本。
- 展示版本号、操作人、时间和原因。
- 支持查看配置 diff。
- 支持回滚到指定历史版本。
- 回滚前展示确认弹窗，包含影响摘要。
- 回滚后刷新当前配置和操作日志。

### 建议交付物

- 管理端配置历史面板。
- 配置 diff 展示组件。
- 回滚确认交互。
- 配置历史相关状态管理。
- 前端构建验证。

### 验收标准

- 管理端可以打开某个配置项的历史列表。
- 管理端可以查看某个历史版本与当前配置的差异。
- 管理端可以触发回滚并看到确认信息。
- 回滚成功后页面显示最新配置。
- 操作日志中可以看到回滚记录。

## 推荐里程碑

### 里程碑一：版本记录打底

先新增配置版本模型、migration 和保存链路版本写入。

交付物：

- `config_versions` 模型。
- migration。
- 保存配置时自动写版本。
- 版本写入测试。

### 里程碑二：历史与 Diff API

补齐历史查询和提交前 diff 预览，让管理端有能力展示变更内容。

交付物：

- 历史列表 API。
- 历史详情 API。
- diff 预览 API。
- JSON diff helper。

### 里程碑三：回滚能力

实现配置回滚，并确保回滚本身可追踪。

交付物：

- 回滚 API。
- 回滚版本记录。
- 回滚操作日志。
- 回滚测试。

### 里程碑四：管理端体验

把历史、diff、回滚串到管理端页面，形成运营闭环。

交付物：

- 配置历史入口。
- diff 展示。
- 回滚确认。
- 回滚后刷新。

## 推荐执行顺序

1. 配置历史模型。
2. 配置 History / Diff API。
3. 配置回滚 API。
4. 管理端配置中心增强。

优先级理由：先确保每次配置变更都有版本依据，再开放查询和 diff 能力；随后实现回滚，最后把能力产品化到管理端页面。这样可以避免先做前端交互但后端缺少可靠版本基础。
