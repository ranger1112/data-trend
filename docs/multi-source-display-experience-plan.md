# 多数据源接入与展示体验完善计划

## 背景

当前项目已经完成数据平台基础能力、部署运行基线、管理端认证权限、采集任务可靠性、数据源扩展机制和小程序 API 契约固化。下一阶段应从“单一房价数据源可跑通”推进到“多数据源可扩展、管理体验可用、小程序展示接近真实产品”。

本阶段围绕四个方向展开：

1. 接入第二个真实数据源。
2. 管理端数据源配置产品化。
3. 小程序页面从 MVP 变成可用展示。
4. 数据质量规则升级。

## 阶段目标

- 用第二个真实数据源验证 importer registry 和数据模型扩展能力。
- 让管理端从工程配置界面升级为可运营的数据源管理界面。
- 让小程序端形成首页、城市详情、趋势和排行的完整展示路径。
- 将质量检查从固定规则升级为按数据源类型配置的规则体系。

## 工作项一：接入第二个真实数据源

### 目标

选择一个与房价数据不同但仍适合公开展示的统计数据源，跑通完整链路：`crawler -> importer -> quality -> admin -> mini API`。

### 候选数据源

- CPI 居民消费价格指数。
- 人口统计数据。
- 就业或失业率数据。
- 地区经济指标，例如 GDP、社零、固定资产投资。

优先选择结构稳定、公开可访问、更新周期明确的数据源。

### 实施内容

- 选定第二个数据源类型，例如 `cpi` 或 `regional_economy`。
- 新增对应 crawler/parser/importer。
- 在 importer registry 中注册新类型。
- 补充指标定义、维度字段和单位。
- 接入现有采集任务、调度和 retry 机制。
- 在管理端展示该数据源类型和最近采集状态。
- 在小程序 API 中确保该数据源可以被指标列表和趋势接口消费。

### 建议交付物

- `packages/crawler/<new_source>/`
- `packages/pipeline/importers.py` 新数据源注册
- 新数据源 parser/importer 测试
- 第二数据源的示例数据或 fixture
- 数据源接入说明文档

### 验收标准

- 第二个真实数据源可以通过管理端创建和触发采集。
- 采集结果进入统一指标、区域和统计值模型。
- 失败、重试和 unsupported type 逻辑不被破坏。
- 小程序 API 可以查询新指标的最新值和趋势数据。
- 测试覆盖 parser、importer、registry 和 API 查询路径。

## 工作项二：管理端数据源配置产品化

### 目标

把管理端的数据源配置从“能填字段”升级为“运营人员可以理解和操作”的管理体验。

### 实施内容

- 数据源类型改为下拉选择，展示已注册 importer 类型。
- 数据源列表展示最近一次采集状态、采集时间、错误原因和成功率。
- 支持调度策略编辑，包括启停、间隔、下一次运行时间。
- 数据源详情页或详情面板展示：
  - 基础信息。
  - 入口 URL。
  - 数据源类型。
  - 最近任务。
  - 质量报告。
  - 手动触发、重试、停用入口。
- 对高风险操作增加确认提示，例如停用数据源、取消运行中任务。
- 优化错误提示，避免直接暴露后端原始异常文本。

### 建议交付物

- admin-web 数据源管理视图重构
- 后端数据源详情 API
- registered importer types API
- 数据源运行状态聚合字段
- 管理端交互和空态优化

### 验收标准

- 管理端可以直观看到每个数据源是否健康。
- 运营人员不需要记忆数据源 type 字符串。
- 数据源调度策略可以在 web 端完成配置和调整。
- 失败任务可以从数据源视角定位和重试。
- 构建通过，关键交互有测试或明确手工验证记录。

## 工作项三：小程序页面从 MVP 变成可用展示

### 目标

基于已固化的小程序 API，将小程序从单页 MVP 升级为可用的数据展示产品。

### 实施内容

- 首页展示总体概览、最新周期、核心指标和热门城市。
- 城市详情页展示城市维度的指标卡片和趋势图。
- 趋势页支持指标、住宅类型、面积类型等筛选。
- 排行榜页支持 top/bottom 切换、指标切换和筛选。
- 城市收藏能力完善，收藏城市在首页优先展示。
- 补充加载态、空态、错误态和重试入口。
- 对接口缓存字段进行利用，减少重复请求。

### 建议交付物

- 小程序首页优化
- 城市详情页
- 趋势图组件或图表封装
- 排行榜页面
- 小程序请求层缓存策略
- 小程序展示验收说明

### 验收标准

- 用户可以从首页进入城市详情、趋势和排行路径。
- 无数据、加载中、请求失败都有明确界面状态。
- 房价指标和第二数据源指标都可以在展示端出现。
- 页面字段只依赖 `docs/miniapp-api.md` 中定义的稳定契约。
- 小程序端主要交互完成真实设备或开发者工具验证。

## 工作项四：数据质量规则升级

### 目标

把当前基础质量检查升级为按数据源类型配置的规则体系，支撑多数据源长期运行。

### 实施内容

- 为每个数据源类型定义质量规则配置。
- 支持基础规则：
  - 必填区域数量。
  - 指标完整性。
  - 周期完整性。
  - 数值范围。
  - 环比、同比异常阈值。
  - 缺失数据标记。
- 质量报告增加明细项，能定位到指标、区域、周期和维度。
- 管理端展示质量报告明细和处理建议。
- 支持按规则严重程度区分 error 和 warning。
- 为质量失败数据保留人工复核和发布阻断流程。

### 建议交付物

- `packages/pipeline/quality.py` 规则配置化
- 数据源类型质量规则定义
- 质量报告明细结构
- 管理端质量报告详情
- 质量规则测试

### 验收标准

- 不同数据源可以使用不同质量规则。
- 质量报告能说明具体哪条规则失败。
- error 级别问题可以阻断发布，warning 级别问题可进入复核。
- 管理端可以查看质量问题明细。
- 测试覆盖完整、缺失、异常值和 warning/error 分级路径。

## 推荐里程碑

### 里程碑一：第二数据源闭环

优先接入第二个真实数据源，验证扩展机制真实可用。

交付物：

- 新数据源 crawler/importer
- registry 注册
- importer 测试
- mini API 查询验证

### 里程碑二：数据源管理体验

在多数据源存在后优化管理端配置和状态展示。

交付物：

- 数据源类型下拉
- 数据源健康状态
- 调度策略编辑
- 失败任务重试入口

### 里程碑三：小程序展示成型

将展示端从单页数据查看升级为完整用户路径。

交付物：

- 首页概览
- 城市详情
- 趋势图
- 排行榜
- 加载、空态、错误态

### 里程碑四：质量规则体系

最后升级质量检查，支撑后续持续新增数据源。

交付物：

- 数据源级规则配置
- 质量报告明细
- 管理端质量详情
- 质量规则测试

## 推荐执行顺序

1. 接入第二个真实数据源。
2. 管理端数据源配置产品化。
3. 小程序页面从 MVP 变成可用展示。
4. 数据质量规则升级。

优先级理由：先用第二数据源验证架构，再让管理端具备多源运营能力；随后补齐小程序真实展示路径；最后将质量规则抽象化，避免后续每新增一个数据源都复制一套检查逻辑。
