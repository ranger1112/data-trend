# 下一阶段生产化闭环计划

## 背景

当前项目已经完成数据平台 MVP：房价数据可以从国家统计局抓取，管理端可以查看和发布数据，小程序端可以按房屋类型和面积类型展示最新值与趋势。下一阶段目标是把 MVP 推进到可长期运行的生产化闭环，而不是继续堆单点功能。

## 阶段目标

本阶段以稳定自动运行和可审核发布为核心，建立从定时抓取、质量校验、人工审核到小程序展示升级的完整链路。

成功标准：

- 系统可以按计划自动抓取数据，并支持失败重试和手动干预。
- 导入后的数据必须经过质量校验，异常数据不会被误发布。
- 管理端具备清晰的审核发布流程和可追踪操作记录。
- 小程序端能用图表和榜单呈现已发布数据，支撑真实用户查看。

## 工作项一：任务调度中心

### 目标

让抓取任务从一次性手动执行升级为可配置、可追踪、可恢复的调度系统。

### 实施内容

- 新增调度配置模型，记录抓取目标、调度周期、启停状态、最近运行时间和下次运行时间。
- 增加定时触发入口，用于周期性创建 `CrawlJob`。
- 支持管理端手动重跑指定任务。
- 支持失败任务重试，记录重试次数和最后失败原因。
- 支持任务取消或禁用，避免异常任务持续占用资源。
- 扩展任务日志，记录开始时间、结束时间、耗时、目标 URL、总记录数、导入记录数、错误类型和错误详情。

### 后端接口建议

- `GET /admin/schedules`：查看调度配置列表。
- `POST /admin/schedules`：创建调度配置。
- `PATCH /admin/schedules/{id}`：启停或修改调度配置。
- `POST /admin/crawl-jobs/{id}/retry`：重试失败任务。
- `POST /admin/crawl-jobs/{id}/cancel`：取消未完成任务。

### 验收标准

- 可以通过调度配置自动创建抓取任务。
- 重复任务不会并发执行同一目标 URL。
- 失败任务可以重试，并保留原失败记录。
- 管理端能看到任务耗时、状态、错误分类和错误详情。

## 工作项二：数据质量校验

### 目标

在数据发布前建立自动质量门禁，避免抓取成功但数据不完整、周期异常或指标异常时进入展示端。

### 实施内容

- 新增数据质量校验服务，对每次导入结果执行规则检查。
- 校验城市数量是否达到预期，例如 70 个城市。
- 校验指标数量和维度组合是否完整。
- 校验最新周期是否合理，避免旧数据覆盖新周期判断。
- 校验同一城市、同一指标、同一周期下是否存在异常重复。
- 校验数值范围，识别空值、非数字、极端异常值。
- 为每次导入生成质量报告，记录通过项、失败项、警告项和关联任务。
- 异常数据保持 draft 或标记为异常，不自动发布。

### 数据状态建议

- `draft`：新导入，尚未审核。
- `quality_failed`：质量校验失败。
- `ready_for_review`：质量校验通过，等待审核。
- `published`：已发布到小程序端。
- `rejected`：人工驳回。

### 验收标准

- 导入完成后自动生成质量报告。
- 城市数、指标数、周期和维度缺失可以被识别。
- 质量失败的数据不会被批量发布接口误发布。
- 管理端可以筛选质量失败和待审核数据。

## 工作项三：审核发布流

### 目标

把当前简单发布动作升级为可审核、可追踪、可回退的管理流程。

### 实施内容

- 管理端增加待审核数据列表。
- 支持按周期、城市、指标、房屋类型、面积类型、质量状态筛选。
- 支持批量发布、批量驳回。
- 支持单条数据编辑，编辑后记录修改前后值。
- 增加发布批次记录，记录发布人、发布时间、发布数量和来源任务。
- 增加驳回原因字段，方便后续复查。
- 小程序端只读取 `published` 数据。

### 后端接口建议

- `GET /admin/review-items`：查看待审核数据。
- `PATCH /admin/stat-values/{id}`：编辑单条数据。
- `POST /admin/review-batches/publish`：批量发布。
- `POST /admin/review-batches/reject`：批量驳回。
- `GET /admin/publish-batches`：查看发布批次历史。

### 验收标准

- draft 数据不能绕过审核直接进入小程序展示。
- 单条编辑有变更记录。
- 发布和驳回都有批次记录。
- 管理端可以追溯某条展示数据来自哪次抓取和哪次发布。

## 工作项四：小程序展示升级

### 目标

把小程序从数据列表升级为可读性更强的数据产品界面。

### 实施内容

- 增加城市搜索，支持快速定位城市。
- 增加常看城市收藏，默认展示收藏城市。
- 增加指标切换控件，支持环比、同比、累计平均。
- 增加趋势图，展示指定城市、指标、房屋类型和面积类型的时间序列。
- 增加涨跌榜，展示最新周期中涨幅靠前和跌幅靠前城市。
- 首页展示最新周期、已发布数据量、重点城市卡片和榜单入口。
- 优化空状态和加载状态，避免接口无数据时页面显得故障。

### 接口建议

- `GET /mini/stat-values/trend`：继续作为趋势图数据源。
- `GET /mini/stat-values/latest`：继续作为最新值数据源。
- `GET /mini/rankings`：新增榜单接口，按指标和维度返回城市排行。
- `GET /mini/cities/search`：可选，城市搜索接口；数据量小也可以前端本地过滤。

### 验收标准

- 用户可以搜索城市并查看趋势图。
- 用户可以切换指标、房屋类型和面积类型。
- 首页能展示最新周期和涨跌榜。
- 小程序只展示已发布数据，不展示 draft 或质量失败数据。

## 建议里程碑

### 里程碑一：自动抓取和质量门禁

优先实现任务调度中心和数据质量校验。

交付物：

- 调度配置模型和管理接口。
- 自动创建抓取任务的 runner。
- 质量校验服务和质量报告模型。
- 管理端展示任务详情和质量报告。
- 覆盖调度、重试、质量失败的测试。

### 里程碑二：审核发布闭环

在质量门禁基础上完善人工审核。

交付物：

- 待审核数据列表。
- 批量发布和批量驳回。
- 单条编辑和变更记录。
- 发布批次历史。
- 小程序只读取已发布数据的回归验证。

### 里程碑三：展示体验升级

在数据链路稳定后增强小程序。

交付物：

- 城市搜索和收藏。
- 趋势图。
- 涨跌榜。
- 首页信息架构优化。
- 小程序端接口 smoke 和核心交互验证。

## 风险与注意事项

- 调度不要一开始引入过重的任务系统；本阶段可以先用轻量 runner 或简单后台进程，等任务复杂度上来后再评估 Celery、APScheduler 或队列方案。
- 质量规则要保留可配置空间，不要把 70 城市、指标组合等规则硬编码到不可调整的位置。
- 审核流要保证小程序只读取 `published` 数据，这是展示可信度的底线。
- 小程序图表不要早于数据质量门禁投入过多精力，否则展示会被脏数据拖累。
- 真实数据验收必须继续使用国家统计局页面，不能只依赖 mock 数据。

## 推荐执行顺序

1. 任务调度中心。
2. 数据质量校验。
3. 审核发布流。
4. 小程序展示升级。

优先级理由：调度和质量校验决定系统能不能长期稳定地产生可信数据；审核流保证可信数据被正确发布；小程序展示升级建立在稳定发布数据之上。
